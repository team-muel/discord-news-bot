## 현재 기준선 (2026-03-15)

아래 항목은 이미 구현/운영 가능한 상태입니다.

- 길드별 Obsidian 지식 트리 자동 bootstrap
- Manifest 기반 재귀 문서 동기화
- All-guilds ops-loop 자동 발견/순회
- Discord topology 스냅샷 자동 반영
- 채널/유저 활동 텔레메트리 자동 반영
- 리액션 보상 신호 자동 반영
- timeout/retry/failure-rate 게이트 및 lock 기반 중복 실행 방지

로드맵의 다음 단계는 위 기준선 위에서 planner/tool routing의 자기개선 루프를 얹는 것입니다.

## 목표

- 디스코드 서버(길드)별 맥락을 장기적으로 학습/참조하는 에이전트 구현
- 게임 레이어 없이 운영/기억/추론 품질을 제품 수준으로 고정
- 기존 Supabase + Render + Vercel 구조를 유지하며 점진적 확장

## 범위 (In)

- 길드 단위 장기기억 데이터 모델
- 맥락 압축 파이프라인 (수집 -> 요약 -> 회수)
- 관리자 교정(수정/고정/폐기) 루프
- 근거 첨부형 응답 규격
- 관측 지표 및 운영 런북

## 범위 (Out)

- 게임 상태머신(마피아, 미니게임)
- 과금/결제 연동
- 고객 로컬 설치형 커넥터

## 아키텍처 원칙

1. 길드 격리 우선: 모든 기억 객체는 guild_id를 필수로 가진다.
2. 근거 우선 응답: 응답은 memory_item_id 또는 source_message_id 근거를 포함한다.
3. 관리자 교정 가능: 자동 생성 기억은 관리자 확정으로 승격 가능하다.
4. 비용 안정성: 압축/회수는 예산과 레이트리밋을 가진다.
5. 실패 가시성: 실패 원인과 복구 힌트가 로그/대시보드에 드러나야 한다.

## 리스크와 대응

- 리스크: 오래된 기억이 최신 정책을 덮어씀
- 대응: 관리자 확정 > 최신성 > 빈도 순 우선순위 적용

- 리스크: 요약 과정에서 맥락 손실
- 대응: 원문 참조 ID를 항상 보존하고 역추적 가능하게 유지

- 리스크: 길드 간 데이터 혼선
- 대응: guild_id + RLS + 테스트 데이터 분리 의무화
